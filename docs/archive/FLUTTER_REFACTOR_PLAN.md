# Flutter 重构方案（归档）

> 这是一份历史规划文档，仅保留作背景参考。  
> 它不描述当前项目的真实实现状态。

## 1. 目标

一句话目标：

将当前项目重构为一款专为 `Z Flip 4` 悬停态设计的桌面端 PC 媒体陪伴控制台，采用 `Flutter` 作为移动端前端，保留 `C#/.NET` 作为 PC 端 Agent，并继续使用 `neteasecli` 作为当前阶段的播放器控制适配层。

核心约束：

- 面向个人使用，不考虑新手引导
- 主场景是半折叠桌搭，不是普通手机 App
- 功能保持克制，只围绕网易云听歌控制
- 不做通用远控平台
- 不做系统全局音量控制

## 2. 当前重构决策

### 2.1 保留的部分

- 保留 PC 端 `ReuseDisplay.Agent`
- 保留 `WebSocket + UDP 广播` 的局域网连接架构
- 保留自包含安装包与分发流程
- 保留 `musiccontrol.json` 白名单动作配置
- 保留 Android 现有工程作为过渡与参考

### 2.2 重构的部分

- 移动端 UI 从当前原生 Android View/XML 重构为 `Flutter`
- 折叠屏主界面改为专用的“上显下控”布局
- 视觉语言统一为蓝白、轻电子、亚克力拟物桌搭风格

### 2.3 当前不改的部分

- 暂不切换到 `SMTC`
- 暂不支持除网易云以外的播放器
- 暂不做复杂系统信息看板扩展
- 暂不做远程桌面、文件管理、消息通知

## 3. 产品定位

一句话定位：

一款专为折叠屏手机设计的桌面端 PC 陪伴型控制台应用。

核心理念：

- 软硬一体的桌搭物件
- 极简低负担的数字陪伴
- 触手可及的媒体控制

视觉基调：

- 蓝白主色调
- 轻赛博 / 轻电子风
- 亚克力拟物感
- 带洛天依气质，但不直接使用角色壁纸

## 4. 目标用户与使用场景

目标用户：

- 拥有固定 PC 桌面
- 注重桌面美学
- 长时间在 PC 前工作 / 游戏 / 听歌
- 手头有 `Z Flip` 系列等折叠屏设备

核心场景：

手机半折叠立于显示器下方或键盘侧边。PC 播放音乐时，用户可余光看到当前播放状态和氛围视觉；无需切换电脑窗口，伸手触碰手机下半屏即可完成播放控制。

## 5. 核心需求定义

### 5.1 上半屏：信息与氛围层

设计原则：

- 高对比度
- 呼吸感
- 远距离可读

需要包含：

- 左上角极简连接光点
- 右上角电池电量线
- 大字号歌名
- 中等字号歌手名
- 进度显示
- 蓝白动态声波 / 粒子背景
- 播放时微动、暂停时呼吸态
- 防烧屏的像素微动或缓慢休眠逻辑

### 5.2 下半屏：控制与交互层

设计原则：

- 盲按
- 大面积
- 实体质感

布局目标：

- 正中央超大 `播放 / 暂停`
- 左侧整块 `上一首`
- 右侧整块 `下一首`
- 角落 `收藏` 或 `静音`
- 音量采用方案 B：
  - 下半屏空白区域双指上下滑动调节
  - 不使用细滑条

反馈要求：

- 所有关键点击具备触觉反馈
- 后续如需提升手感，可补 Android 原生马达调用

### 5.3 系统与连接层

要求：

- 局域网自动发现与自动连接
- 手动 IP 作为兜底，不作为主路径
- App 保持常亮
- 前台服务稳定运行

## 6. 非核心需求与明确不做

V1 不做：

- PC 远程桌面或屏幕镜像
- 文件管理
- 复杂系统监控扩展
- 社交与消息推送
- 多级菜单
- 通用命令执行平台
- 多播放器兼容

## 7. 技术架构

### 7.1 PC 端

保留：

- `C#/.NET`
- 常驻 Agent
- `WebSocket + UDP`
- `musiccontrol.json`

继续使用：

- `neteasecli`

动作模型：

- `music.playPause`
- `music.next`
- `music.previous`
- `music.stop`
- `music.like`

后续可选：

- `music.volumeUp`
- `music.volumeDown`

前提是 CLI 能提供稳定的播放器内音量能力。

### 7.2 Flutter 端

建议分层：

1. `network`
   - UDP 自动发现
   - WebSocket 长连接
   - 手动 IP 兜底
   - 重连策略

2. `protocol`
   - 状态消息
   - 命令消息
   - 命令结果消息

3. `state/controller`
   - 连接状态
   - 音乐状态
   - 折叠姿态状态
   - 页面模式状态

4. `ui`
   - 上半屏陪伴层
   - 下半屏控制层
   - 设置页

原则：

- Flutter 端不感知 `neteasecli` 细节
- Flutter 端只发动作 ID
- Flutter 端只消费状态数据

## 8. 风险分析与处理方案

### 8.1 风险 1：折叠姿态能力

修正后的结论：

- 优先尝试纯 Flutter
- 使用 `MediaQuery.of(context).displayFeatures`
- 验证是否能正确识别 `FoldingFeature`、`HALF_OPENED` 和折痕 bounds
- 如验证失败，再补 Android 原生桥接

执行策略：

1. Flutter 新工程初始化后，第一时间做折叠姿态验证页
2. 在真实 `Z Flip 4` 上验证半折叠表现
3. 只有失败时才写原生桥接

### 8.2 风险 2：neteasecli 音量控制

风险点：

- 很多 CLI 只支持增量音量，不支持绝对值
- 高频滑动不适合直接高频发送命令
- 可能无法稳定获取当前播放器内音量值

处理方案：

- Flutter 端维护本地“假音量状态”
- 手势过程中只更新本地 UI
- `onEnd` 或停顿 `300ms` 后再提交最终命令
- 采用“UI 连续、执行离散”的策略

设计原则：

- V1 不强依赖精确音量百分比
- 更偏重顺手感，而不是精确同步

### 8.3 风险 3：触觉反馈质感不足

阶段策略：

- 先用 Flutter 自带 `HapticFeedback`
- 如果后续手感不满意，再补 `MethodChannel`
- 调用 Android 原生 `VibrationEffect.createPredefined(...)`

当前结论：

- 不提前复杂化
- 先以产品形态和交互完成度为主

## 9. 迁移策略

总原则：

- 先并行
- 后切换
- 最后清理

### 9.1 保留旧 Android 工程的原因

- 当前已跑通连接链路
- 当前已跑通前台服务 / 常亮 / 手动 IP 等能力
- 旧工程可作为 Flutter 重构时的参考实现

### 9.2 旧工程后续可废弃内容

- XML 布局
- 当前 `MainActivity` 页面结构
- 当前 View/UI 状态层
- 当前主题资源

### 9.3 旧工程后续可保留参考内容

- 权限声明
- 设备管理员 / 启动恢复思路
- WebSocket / UDP 交互逻辑
- 签名与构建脚本经验

## 10. 最终执行顺序

### 阶段 1：冻结 PC 协议与动作边界

- 固定 WebSocket 消息格式
- 固定动作白名单
- 固定 `musiccontrol.json` 结构
- 固定 `neteasecli --json player status` 解析

### 阶段 2：初始化 Flutter 前端

- 在仓库中新建 Flutter 客户端
- 先不动 PC 端主体
- 跑通 Android 构建

### 阶段 3：验证折叠姿态

- 使用 `MediaQuery.displayFeatures`
- 验证 `Z Flip 4` 半折叠下能否正确识别折痕与状态
- 决定是否需要原生桥接

### 阶段 4：跑通最小链路

- 自动发现
- WebSocket 连接
- 状态显示
- 核心按钮控制

### 阶段 5：实现半折叠布局

- 上半屏陪伴层
- 下半屏控制层
- 单一主界面

### 阶段 6：实现音量方案 B

- 双指上下滑动
- 本地音量 HUD
- `onEnd / debounce` 提交

### 阶段 7：实现触觉反馈

- Flutter 内置震动先接入
- 关键按钮与手势都具备反馈

### 阶段 8：注入视觉灵魂

- 蓝白冰透风格
- 轻电子动效
- 洛天依气质抽象化
- 微拟物按压感

### 阶段 9：清理与切换

- Flutter 端成为主前端
- 旧 Android UI 退场
- 更新分发包与说明文档

## 11. 当前推荐结论

最终结论：

- 现在迁 Flutter 是合理的
- 当前继续保留 `C# Agent + neteasecli` 是合理的
- `SMTC` 暂不作为当前阶段重点
- 音量控制先遵循“本地顺手，远端离散执行”的原则
- 项目应继续收敛为个人桌搭音乐终端，而不是通用远控平台
